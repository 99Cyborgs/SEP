# Support Navigation Evidence

## Purpose
Treat the support-portal benchmark as a bounded cross-domain application slice rather than a theorem about all help or support navigation systems.

## What Is Held Fixed
- the four-page support funnel state set
- the ordered support-lifecycle windows
- the named-window operator construction and pseudocount refinement protocol
- the transport-style acceptance criteria already used for non-mobility navigation evidence

## What Is Varied
- one accepted stable support funnel
- one detour-heavy rejection slice with gap collapse, strong carrier deformation, and explicit failure labels

## Interpretation
- `reference` is accepted only as a bounded support-navigation case study
- `negative_detour` is retained as rejection evidence, not tuned into acceptance
- the negative slice is rejected by the benchmark-local package overlay because its singular gap and carrier geometry both miss the acceptance floor; its residual coupling worsens but does not cross the repository-wide `coupling_failure` cutoff
- the intended claim is structural only: the operator workflow can transfer beyond mobility and one documentation clickstream benchmark

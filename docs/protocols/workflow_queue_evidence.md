# Workflow Queue Evidence

## Purpose
Treat the workflow-queue benchmark as a bounded operational case study rather than a theorem about generic service or work-management systems.

## What Is Held Fixed
- the four-stage workflow state set
- the ordered intake-to-closure windows
- the named-window operator construction and pseudocount refinement protocol
- the transport-style acceptance criteria already used for bounded application evidence

## What Is Varied
- one accepted stable queue with persistent intake-versus-execution separation
- one rework-heavy rejection slice with explicit carrier degradation and weaker stage separation

## Interpretation
- `reference` is accepted only as a bounded workflow-queue case study
- `negative_detour` is retained as rejection evidence, not tuned into acceptance
- the negative slice is rejected by the benchmark-local package overlay because its singular gap and carrier deformation miss the declared workflow floors; the residual coupling is weaker than the reference case but stays below the repository-wide `coupling_failure` cutoff
- the intended claim is structural only: the operator workflow can transfer beyond mobility and page-navigation fixtures into a non-navigation operational queue
